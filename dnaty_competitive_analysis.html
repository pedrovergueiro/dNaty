Resumo

dNATY é uma biblioteca open-source de compressão evolutiva de redes neurais MLP/CNN voltada para inferência em hardware sem GPU. A versão 2.0 introduz busca evolutiva orientada a latência real (ONNX Runtime), estimativa de latência em dispositivos alvo sem hardware físico (RPi 4/5, Jetson Nano), esparsidade estruturada N:M (2:4), quantização INT8 integrada ao processo de busca, e monitoramento de produção com auto-retriggering de compressão. Em 11 datasets industriais avaliados, a redução de FLOPs varia de 36% a 83% com acurácia mantida ou melhorada em todos os casos. Em aprendizado contínuo (Split-MNIST, 5 tarefas, 3 seeds), o dNATY apresenta 4,9× menos esquecimento catastrófico que EWC (Backward Transfer: −0,204 vs −0,998).

Palavras-chave: neural architecture search, model compression, edge inference, evolutionary algorithms, NSGA-II, continual learning, Pareto optimization, IoT, CPU-only deployment

1. Posicionamento e gap de mercado

O deploy de modelos de aprendizado de máquina em dispositivos embarcados — câmeras industriais, drones autônomos, sensores IoT, robôs — enfrenta um gargalo estrutural: as ferramentas de compressão existentes exigem GPU, retreinamento completo, ou integração profunda com o pipeline de treino original. O dNATY endereça exclusivamente o cenário em que nenhuma dessas três condições pode ser satisfeita.

Proposição central
Dado um modelo PyTorch já treinado (qualquer nn.Module), o dNATY retorna um modelo comprimido que roda em CPU genérica — sem acesso ao pipeline de treino original, sem GPU, sem retreinamento, com uma única chamada de função.

2. Funcionalidades — dNATY v2.0

2.1 API principal

from dnaty import compress, load

# Compressão por FLOPs (padrão)
result = compress(model, X_train, y_train, target_flops=0.5)

# Compressão orientada a latência em RPi 4 (sem ter o hardware)
result = compress(model, X, y, target="latency", hw_target="rpi4")

# Com esparsidade 2:4 + quantização INT8
result = compress(model, X, y, sparsity="2:4", quant_aware=True)

# Aceita numpy, pandas ou DataLoader — sem conversão manual
result = compress(model, df)  # DataFrame: última coluna = label

result.summary()              # métricas de compressão
result.save("model.pt")      # persiste arquitetura + pesos
result.export_onnx("model.onnx", input_shape=(784,))
result.benchmark_latency((784,))   # p50/p95/FPS em CPU
q = result.quantize()         # INT8 dinâmico em uma linha

2.2 Módulos e evolvers disponíveis

DnatyEvolver
NAS evolutivo base com NSGA-II. Otimiza acurácia × FLOPs. Episodic memory pondera operadores bem-sucedidos em gerações anteriores.
LatencyEvolver
NAS orientado a latência real (ONNX Runtime). Usa surrogate GBM para 80% das avaliações; medição real para 20%. Retorna Pareto front acurácia × latência.
CnnEvolver
NAS para CNNs com seleção FLOPs-guided de camadas. Budget-aware boost (3×) quando o modelo excede o budget de FLOPs. Validado em CIFAR-10.
QuantAwareEvolver
NAS com fitness avaliado sob quantização INT8 dinâmica. Arquiteturas que degradam sob quantização são filtradas durante a busca. +20% overhead vs base.
compress_with_backbone
Suporte a ResNet, MobileNetV2, EfficientNet, ViT. Extrai features do backbone congelado, aplica NAS na cabeça MLP, reconecta. finetune_backbone=True disponível.
DriftDetector + ProductionTracker
Monitoramento PSI/KL em produção. auto_retrigger recomprime automaticamente após N drifts consecutivos. export_failure_report com PCA 2D e SQLite.
apply_nm_sparsity
Esparsidade estruturada N:M (padrão 2:4 — 50% zeros). Pesos baked no .data, sem hooks. Compatível com export ONNX e inferência INT8.
LatencyPredictor
Surrogate GBM treinado em (features de arquitetura → latência ms). Degrada graciosamente para fallback analítico sem LightGBM. Flywheel de dados via telemetria.
detect_hw / estimate_latency
Detecção de CPU (x86_64, ARM). Tabelas calibradas para RPi 4 (9×), RPi 5 (4,5×), Jetson Nano (5×), Apple M1 (0,6×). Estima latência sem ter o hardware.
count_flops / flops_by_layer
Contagem precisa de FLOPs por operação: nn.Linear, nn.Conv2d, depthwise separável. Usado internamente pelo NSGA-II e exposto na API pública.

Vantagens reais do dNATY
  • CPU + pós-treino + sem retreinamento: combinação única — nenhum concorrente entrega os três
  • Latency NAS sem hardware físico: estima latência em RPi/Jetson a partir de medição x86
  • INT8 integrado ao NAS: arquiteturas ruins sob quantização são filtradas durante a busca
  • CL out-of-the-box: nenhuma ferramenta concorrente oferece replay nativo
  • Monitoramento de produção: drift detection + auto-retrigger integrados
Limitações honestas
  • CNN profunda direta: TensorRT e NNCF são mais maduros para ResNet/EfficientNet sem backbone
  • Latência estimada (não medida): fatores de escala para RPi/Jetson têm ±30% de erro típico sem hardware real
  • GBM surrogate: requer pip install lightgbm opcional; sem ele usa fallback analítico
  • Foco MLP/tabular: NAS direto em CNNs profundas está no roadmap v2.x

5. Perguntas técnicas frequentes

Por que não usar TFLite ou TensorRT?
TensorRT exige hardware NVIDIA — dispositivos ARM com CPU genérica não são suportados. TFLite requer acesso ao pipeline de treino original e normalmente quantization-aware training; não é possível partir de um .pth qualquer sem retreinar. O dNATY opera exclusivamente pós-treino.
Por que não usar NNCF da Intel?
NNCF é uma ferramenta de treino, não de pós-treino — requer hooks no loop de loss e normalmente GPU recomendada. O dNATY não precisa do código de treino original, apenas do modelo e de uma amostra de dados de validação.
Por que não OFA ou MnasNet?
OFA treina um supernet do zero por dias em cluster de GPU e então extrai subnets. São propostas diametralmente opostas: dNATY parte de um modelo já treinado. Custo de entrada: OFA = semanas de GPU + dataset completo de treino. dNATY = minutos de CPU + amostra de validação.
dNATY funciona com CNNs?
compress_with_backbone suporta ResNet, MobileNetV2, EfficientNet e ViT — extrai features do backbone congelado, aplica NAS na cabeça MLP e reconecta. NAS estrutural direto em CNNs profundas (camada a camada) está no roadmap v2.x. Para o caso de uso do CEIA (câmeras, drones), o fluxo backbone + MLP cobre a maioria dos cenários reais.
Os resultados são reproduzíveis?
Sim. O repositório contém 72 testes automatizados (pytest) e scripts de benchmarking com dados reais do Kaggle. O script python scripts/prove_it.py reproduz os resultados de NAS e aprendizado contínuo com seed fixo. Todos os datasets utilizados são públicos com instruções de download no README.
Como o dNATY se posiciona academicamente?
A abordagem NSGA-II para NAS é bem estabelecida na literatura. A contribuição do dNATY é a composição: (1) NAS multiobjetivo pós-treino em CPU; (2) otimização Pareto de latência real via ONNX Runtime + surrogate GBM; (3) esparsidade N:M integrada à busca evolutiva; (4) aprendizado contínuo com replay nativo. Nenhuma ferramenta existente integra essas quatro dimensões em uma única API.

6. Proposta de colaboração — CEIA-UFG

O alinhamento com as linhas de pesquisa do CEIA em visão embarcada e sistemas autônomos abre três frentes complementares:

A · Teste de campo
Aplicar o dNATY em modelos já treinados do CEIA para câmeras, drones ou sensores. Medir ganho real de latência no hardware do grupo. Resultado mensurável em dias. Contribuição do CEIA: hardware IoT e modelo existente.
B · Parceria de pesquisa
Co-autoria em publicação sobre NAS evolutivo para edge — contribuição técnica do dNATY (código, benchmarks, API) + expertise do CEIA em visão e robótica. Venue potencial: BRACIS, ENIAC, IJCNN, IEEE Access.
C · Troca técnica
Sessão técnica de 20–30 min com o grupo. Apresentação da abordagem evolutiva, discussão de limitações e extensões. Sem compromisso formal. Ponto de partida natural para A ou B.
